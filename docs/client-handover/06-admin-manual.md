# 6. Руководство администратора

Под **администратором** понимаются:

- **Администратор организации** — управление пользователями и настройками внутри своей компании в кабинете (при наличии прав).
- **Администратор платформы (admin / super_admin)** — доступ к API раздела `/admin` и расширенным операциям по пользователям и компаниям.

Сценарии администрирования интеграций с внешними торговыми площадками в данный документ **не** входят.

---

## 6.1. Администратор организации (кабинет)

### Пользователи

1. Войдите в кабинет под учётной записью с правами управления пользователями.
2. Откройте раздел **Пользователи** (или **Команда**).
3. Пригласите сотрудника по e-mail или создайте учётную запись — в соответствии с доступным UI.
4. Назначьте роли так, чтобы ограничить доступ к чувствительным разделам (документы, настройки интеграций).

### Интеграции и API-ключи

1. Раздел **Интеграции**.
2. Выпуск и **отзыв** API-ключей — только для доверенных лиц; при компрометации ключ немедленно отозвать и выпустить новый.
3. Настройка URL **webhook** — HTTPS, доступный из сети сервера приложения.

### Настройки компании

Профиль организации, реквизиты, при необходимости — лимиты по тарифу (отображение зависит от продукта).

---

## 6.2. Администратор платформы (роли admin / super_admin)

Доступ к функциям через REST API префикса **`/admin`** (не путать с UI кабинета). Типовые возможности (см. `backend/app/routers/admin.py`):

| Функция | Описание |
|---------|----------|
| Список пользователей | Просмотр пользователей платформы (super_admin) |
| Создание / блокировка пользователей | Управление учётными записями |
| Связь пользователь ↔ компания | Назначение пользователя в организацию |
| Вход от имени пользователя (impersonation) | Если реализовано — для поддержки, с аудитом |

**Аутентификация:** JWT пользователя с ролью `admin` или `super_admin` (получение токена через `/auth/login`).

Подробные пути и тела запросов — в OpenAPI (`/docs`), тег `admin`.

### Создание первого super_admin

При первичном развёртывании используйте процедуру из **`deploy/DEPLOY_GUIDE.md`** (скрипт создания супер-админа или SQL-вставка) — не храните пароли в документации.

---

## 6.3. Администратор инфраструктуры (сервер)

Относится к документу **«Инструкция по установке и эксплуатации»**:

- управление systemd-сервисами backend и worker;
- nginx, SSL, резервное копирование;
- переменные `.env`, права на `STORAGE_PATH`;
- миграции БД и обновления версии.

---

## 6.4. Безопасность

- Минимизируйте количество учётных записей с ролью `super_admin`.
- Включите двухфакторную аутентификацию на сервере и в панели хостинга, если доступна.
- Регулярно проверяйте список активных API-ключей и сессий (если в UI есть раздел сессий).
- Журналы аудита (если включены) периодически выгружайте для комплаенса.

---

## 6.5. Роль «торговый представитель» (trader)

При включённом сценарии дистрибуции часть пользователей видит **упрощённый контур** кабинета (отдельные маршруты `trader/*`). Настройка лимитов и тарифов — на стороне backend и админ-процедур; детали см. в коде и миграциях. Синхронизация с внешними площадками в руководство не входит.
