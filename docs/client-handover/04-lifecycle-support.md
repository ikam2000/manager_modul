# 4. Техническая документация по поддержанию жизненного цикла ПО

Документ описывает, как поддерживать продукт **ikamdocs** в эксплуатации: версии, окружения, обновления, мониторинг и ответственность сторон. Синхронизация с внешними торговыми площадками в объём настоящей поставки **не** входит.

---

## 4.1. Модель жизненного цикла

| Фаза | Содержание |
|------|------------|
| **Разработка** | Изменения в ветках Git, code review по политике заказчика |
| **Сборка** | Backend: зависимости из `requirements.txt`; Frontend: `npm run build` → артефакты в `frontend/dist/` |
| **Тестирование** | Рекомендуется стенд (отдельная БД и домен), smoke-тесты API и UI |
| **Внедрение** | Деплой на продакшен по `deploy/DEPLOY_GUIDE.md` и `deploy/deploy.sh` |
| **Эксплуатация** | Мониторинг доступности, бэкапы, реакция на инциденты |
| **Вывод из эксплуатации** | Архивирование данных, отзыв ключей и доступов |

---

## 4.2. Окружения

Рекомендуемая схема:

| Окружение | Назначение |
|-----------|------------|
| **dev** | Локальная разработка, SQLite или отдельная БД |
| **stage** | Проверка миграций и сборки перед продакшеном |
| **prod** | Рабочая среда пользователей |

Для каждого окружения — **отдельный** `.env`, отдельная БД и секреты. Не использовать продакшен-ключи на стендах.

---

## 4.3. Версионирование и совместимость

- **Backend:** версия API в OpenAPI (`app/main.py`); при изменении контрактов REST — обновление клиентов интеграций.
- **База данных:** схема версионируется SQL-миграциями в `backend/migrations/` и учётом в `schema_migrations` через `migrate.sh`.
- **Frontend:** сборка привязана к commit; кэш браузера для `index.html` отключён заголовками nginx в типовом конфиге — после деплоя пользователи получают новую SPA-оболочку.

Обратная совместимость миграций: новые миграции должны быть идемпотентны по политике проекта или применяться строго по порядку.

---

## 4.4. Процесс релиза (рекомендуемый)

1. Зафиксировать изменения в Git (тег релиза опционально).
2. Прогнать миграции на stage.
3. Собрать frontend и проверить критичные страницы кабинета.
4. Бэкап prod БД и storage.
5. Деплой на prod, перезапуск сервисов.
6. Проверка health, входа, одного сценария записи данных.

---

## 4.5. Мониторинг и наблюдаемость

- **Доступность:** HTTP health backend, ответ nginx.
- **Логи:** `journalctl` для unit systemd; при необходимости централизованный сбор (ELK, Loki и т.д.).
- **Метрики:** при включённом Prometheus — сбор с эндпоинта приложения.
- **Диск и БД:** алерты по заполнению диска и размеру БД.

---

## 4.6. Безопасность в жизненном цикле

- Регулярные обновления ОС и зависимостей (`pip`, `npm audit`).
- Ротация `SECRET_KEY`, API-ключей и паролей БД при компрометации или уходе сотрудников.
- TLS на границе (nginx + Let's Encrypt или корпоративный сертификат).
- Ограничение доступа SSH и админ-панелей по IP/VPN — по политике заказчика.

---

## 4.7. Зависимости от внешних сервисов

| Сервис | Влияние на ЖЦ |
|--------|----------------|
| PostgreSQL | Критично; резервирование и обслуживание — зона ответственности эксплуатации |
| Redis | При недоступности возможны ошибки сессий/кэша — мониторинг |
| SMTP | Влияет на регистрацию и уведомления |
| Платёжный провайдер | Влияет на оплату; webhook URL должен быть стабильным |
| DaData (если используется) | Опциональные подсказки по контрагентам |

Интеграции с торговыми площадками в данном документе не рассматриваются.

---

## 4.8. Документирование изменений

Рекомендуется вести `CHANGELOG.md` или релиз-ноты в Git: дата, номер версии, миграции БД, инструкции по обновлению.
