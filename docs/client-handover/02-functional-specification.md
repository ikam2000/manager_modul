# 2. Описание функциональных характеристик ПО

**Продукт:** ikamdocs — веб-платформа для управления номенклатурой, справочниками контрагентов, документами и QR-маркировкой в корпоративном контуре.

**Примечание:** настоящее описание **не** включает сценарии синхронизации с внешними торговыми площадками (маркетплейсами). При необходимости такие возможности оформляются отдельной поставкой или доработкой.

---

## 2.1. Назначение системы

Обеспечить организациям единое веб-пространство для:

- ведения каталога товаров и связанных справочников;
- учёта поставщиков, производителей, договоров и поставок;
- хранения и обработки прикреплённых документов;
- выпуска и учёта QR-кодов на номенклатуру;
- обмена данными с внутренними системами (1С, ERP, CRM) через REST API с ключом доступа;
- разграничения доступа пользователей по ролям и компаниям (мультитенантность).

Маркетинговый сайт и личный кабинет реализованы как SPA (React); серверная часть — REST API на FastAPI.

---

## 2.2. Какие задачи решает

| Задача | Как решается |
|--------|----------------|
| Единый каталог номенклатуры | Сущности: категории, подкатегории, карточки SKU, связи с поставщиками и документами |
| Учёт контрагентов и договоров | Поставщики, производители, заказчики, договоры и приложения, поставки |
| Документооборот по товарам | Загрузка файлов, привязка к сущностям, извлечение данных (в т.ч. с поддержкой OCR при настройке окружения) |
| Маркировка и прослеживаемость | Генерация и печать QR, просмотр связанной информации по коду |
| Интеграция с учётными системами | API с ключом `X-Api-Key`: создание и обновление сущностей извне |
| Монетизация и лимиты | Тарифы, оплата через платёжный провайдер (при настройке), учёт лимитов по плану |
| Аналитика | Дашборды и отчёты по данным компании |
| Безопасность и аудит | JWT-аутентификация, роли, журнал действий (где предусмотрено) |

---

## 2.3. Основные модули

| Модуль | Назначение |
|--------|------------|
| **Публичный сайт** | Маркетинговые страницы, регистрация/вход, формы обращения |
| **Аутентификация** | Регистрация, вход, обновление токенов, профиль, сброс пароля (при настроенной почте) |
| **Личный кабинет** | Рабочий интерфейс компании: дашборд, сущности, документы, QR, пользователи, настройки, интеграции (API-ключи, webhooks), уведомления, обращения в поддержку |
| **Сущности (entities)** | CRUD по номенклатуре, категориям, поставщикам, производителям, поставкам, договорам и др. |
| **Документы** | Загрузка, список, привязка к объектам, импорт данных из документов |
| **QR** | Выдача данных по коду, сценарии печати этикеток |
| **Интеграционный API** | `/api/v1/integrate` — машинный доступ для внешних систем |
| **Аналитика** | Отчёты и показатели по компании |
| **Платежи** | Создание платежей, webhook провайдера (при подключении ЮKassa) |
| **Администрирование платформы** | Управление пользователями и компаниями для ролей admin / super_admin |
| **Фоновые задачи** | Планировщик (например, периодические задачи по данным поставщиков — при включённой конфигурации) |
| **Роль «торговый представитель» (trader)** | Отдельный контур интерфейса и лимитов для сценариев дистрибуции (настройка и включение — по политике внедрения) |

Инфраструктурно: reverse proxy (nginx), ASGI-приложение (Uvicorn), PostgreSQL, при необходимости Redis.

---

## 2.4. Ключевые пользовательские сценарии

1. **Регистрация и создание компании** — пользователь регистрируется, получает доступ к кабинету организации.
2. **Ведение номенклатуры** — создание категорий, карточек товаров, заполнение атрибутов.
3. **Работа с поставщиками и договорами** — ввод контрагентов, договоров, фиксация поставок.
4. **Загрузка документов** — файл прикрепляется к номенклатуре или другой сущности; при необходимости выполняется разбор содержимого.
5. **Выпуск QR** — формирование кодов для выбранных позиций, печать этикеток.
6. **Просмотр по QR** — сканирование кода ведёт к данным карточки (в рамках прав доступа).
7. **Интеграция учётной системы** — администратор кабинета выпускает API-ключ и настраивает обмен данными по REST.
8. **Аналитика** — пользователь просматривает отчёты и показатели по своей компании.
9. **Оплата тарифа** — при подключённом платёжном модуле пользователь оплачивает подписку или разовый платёж.
10. **Администрирование пользователей в компании** — присоединение сотрудников, роли (в рамках политики продукта).

Сценарии синхронизации каталога с внешними торговыми площадками в данный документ **не** входят.

---

## 2.5. Что получает пользователь на выходе

- Структурированные данные в единой базе (номенклатура, контрагенты, документы).
- Файлы документов на сервере в каталоге хранения (путь задаётся конфигурацией).
- Печатные формы и этикетки с QR (через интерфейс кабинета).
- Ответы API и вебхуки для внешних систем (при настройке).
- Подтверждения оплат и статусы подписки (при использовании платежного модуля).
- Уведомления по e-mail (при настроенном SMTP).

---

## 2.6. Ограничения и область применения

**Область применения:** B2B, учёт товарных данных и сопутствующей документации внутри организации и обмен с собственными системами учёта.

**Ограничения (типовые):**

- Производительность и масштаб зависят от выделенных ресурсов сервера и объёма БД.
- Качество распознавания документов зависит от качества исходных файлов и настроек OCR (если используется).
- Доставка e-mail зависит от внешнего SMTP и политик антиспама.
- Интеграционный API требует разработки со стороны внешней системы по описанию OpenAPI (`/docs`).
- Функции, завязанные на опциональные ключи API (поиск по реквизитам контрагентов, AI и т.д.), недоступны без их настройки.
- **Не входит в настоящее описание:** автоматическая выгрузка/загрузка каталога на внешние торговые площадки; при наличии кода таких модулей в репозитории их включение и сопровождение определяются отдельным соглашением.

**Юридический режим персональных данных и отраслевые требования** определяются политикой заказчика и договором на обработку данных; продукт предоставляется как инструмент хранения и обработки данных на инфраструктуре заказчика или подрядчика по согласованию.
